# 研究計畫書：OpenClaw GitHub 發布後 SaaS 企業受衝擊程度與企業特徵研究

## 1. 研究背景與動機
隨著 OpenClaw 這類開源 AI agent 平台進入市場，投資人與 SaaS 經營團隊開始重新評估哪些軟體公司的價值來自可替代的流程封裝，哪些則來自較難被快速複製的資料、工作流嵌入與企業級交付能力。

本研究不以嚴格的因果推論為目標，而是聚焦於一個較可操作的問題：**在 OpenClaw GitHub 首次公開發布之後，哪些美國上市 SaaS 企業出現較明顯的市場或基本面調整，且這些企業具備哪些共同特徵。**

## 2. 研究問題
本研究將回答以下三個核心問題：

1. OpenClaw GitHub 首次公開發布後，哪些 SaaS 企業在短期市場反應上顯示較高衝擊程度。
2. 這些企業在成長、獲利、成本結構與估值上具有哪些共同特徵。
3. 以公開財報與市場資料建立的特徵表，是否足以形成具商業解釋力的企業集群。

## 3. 研究範圍與事件定義
### 3.1 研究對象
研究對象為在事件窗口前後具公開財報與股價資料的美國上市 SaaS 公司。第一版將先以 12-15 家驗證樣本進行可行性測試，再於正式報告擴展至 20-30 家候選公司。

### 3.2 事件定義
本研究將 **OpenClaw 首次發布到 GitHub 的日期** 定義為唯一事件日。

- 事件日欄位：`openclaw_github_launch_date`
- 狀態：`待驗證`
- 驗證來源：GitHub repository 建立日期、最早公開 release 或最早可追溯公開 commit 與官方文件交叉比對

若後續驗證發現 GitHub 首發日無法形成明確市場事件，正式報告將保留同一日期作為研究錨點，但在結論中將其表述為「技術公開節點」而非「市場明確認知事件」。

## 4. 研究設計
### 4.1 研究定位
本研究為**描述型與關聯型分析**，目的在於辨識受衝擊企業的輪廓與特徵，不主張直接證明 OpenClaw 對個別公司產生因果效果。

### 4.2 衝擊程度衡量
研究將分成兩層衝擊指標，避免短期市場反應與基本面變化混為一談。

#### A. 市場衝擊指標
- 事件窗累積報酬率
- 相對基準指數的超額報酬
- 事件後估值壓縮幅度
- 事件後股價波動度變化

#### B. 基本面衝擊指標
- 營收成長率變化
- 毛利率變化
- 營業利益率變化
- 研發費用率與銷售費用率變化

正式分析可分別呈現市場衝擊、基本面衝擊與綜合衝擊排序，但 2 週驗證版以**市場衝擊指標**為主。

### 4.3 企業特徵表
為了確保兩週內可落地，主分析只使用公開財報與公開市場資料中一致性較高的欄位。

#### 核心欄位
- `revenue_growth`
- `gross_margin`
- `operating_margin`
- `rd_to_revenue`
- `sm_to_revenue`
- `fcf_margin`（若可穩定取得）
- `ps_ratio`
- `market_cap`

#### 暫不納入主分析的欄位
- NDR
- CAC
- 非標準化產品使用量指標
- 需依賴商業資料庫才能穩定取得的欄位

### 4.4 方法
#### 方法一：事件窗市場反應分析
以 OpenClaw GitHub 首發日為中心，建立單一事件窗，觀察各公司股價報酬、超額報酬與波動度變化。2 週驗證版只使用一套事件窗設定，以降低方法膨脹風險。

#### 方法二：集群分析
不再預先定義 A/B 組，而是對公司特徵表做標準化後，以非監督式分群方法形成企業集群，再比較各群的市場衝擊表現與特徵輪廓。

#### 方法三：關聯分析
使用截面回歸或相關分析，檢查哪些財務特徵與衝擊程度較相關。結果僅作關聯解讀，不使用因果措辭。

#### 方法四：輔助文本分析
正式報告可加入 earnings call 關鍵字觀察，檢視管理層是否更常提及 AI、automation、efficiency、pricing 等主題。2 週驗證階段僅抽取 3-5 家代表公司做人工作為輔助閱讀，不進行情緒分析。

## 5. 可行性驗證設計
### 5.1 兩週驗證版目標
兩週內的目標不是完成完整研究，而是完成一個可重現的 prototype，驗證以下三件事：

1. OpenClaw GitHub 首發日是否能作為可操作的研究錨點。
2. 公開資料是否足夠建立穩定的公司特徵表與市場衝擊指標。
3. 集群分析是否能產出具解釋力的企業分群。

### 5.2 兩週驗證版範圍
- 樣本數：12-15 家
- 指標數：6-8 個核心欄位
- 方法數：單一事件窗分析 + 單一版本集群分析
- 文本分析：不納入主體，只做少量代表公司輔助觀察

### 5.3 兩週驗證完成標準
- 有一份可重現的樣本資料表
- 有至少一套可解釋的集群結果
- 有至少一個可用的市場衝擊衡量方式
- 能明確說明正式報告哪些範圍可做、哪些不建議做

## 6. 預期貢獻
1. 建立 AI agent 衝擊下 SaaS 企業的特徵式分析框架，而非依賴主觀分組。
2. 提供投資研究與產業研究可重複使用的公司特徵表與衝擊評估方法。
3. 協助辨識哪些 SaaS 公司較容易面臨估值重定價與成長放緩風險。

## 7. 研究限制
- 研究錨點為 OpenClaw GitHub 首發日，但市場未必將其視為單一明確事件。
- 公開財報欄位揭露不完全一致，可能降低有效樣本數。
- 集群分析可能在統計上成立，但仍需透過商業語境解釋其含義。
- 本研究定位為描述與關聯分析，不直接主張因果關係。

## 8. 研究時間表
### 第一階段：2 週可行性驗證
- 第 1-2 天：確認事件日、敲定驗證樣本、建立欄位字典與資料來源規則
- 第 3-6 天：蒐集股價與公開財報數據，建立特徵表並記錄缺值
- 第 7-9 天：建立市場衝擊指標並完成一次集群分析
- 第 10-12 天：比較各群事件後表現，歸納 2-3 類代表公司輪廓
- 第 13-14 天：完成 feasibility memo，決定正式報告保留與刪除項目

### 第二階段：2 個月正式報告
- 第 1-2 週：完成 feasibility 驗證並鎖定正式設計
- 第 3-5 週：擴樣本、補齊資料、固定特徵欄位
- 第 6-7 週：完成主要分析、穩健性檢查與代表案例解讀
- 第 8 週：撰寫完整報告、圖表與研究限制

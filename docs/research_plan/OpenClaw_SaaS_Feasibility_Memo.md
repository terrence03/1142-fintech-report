# OpenClaw SaaS 研究可行性備忘錄

**版本**：v1  
**最後更新**：2026-04-16  
**用途**：作為 2 週驗證與 2 個月正式報告的執行備忘錄

## 1. 核心判斷
在公開資料為主、不依賴商業資料庫的前提下，本研究具備可行性，但必須明確降低範圍。最合理的路徑是先完成一個小樣本、可重現、可解釋的 prototype，再決定是否擴樣。

整體判斷如下：

- **2 週驗證**：中高可行
- **2 個月完稿**：可行，但必須固定樣本、欄位與方法

## 2. 2 週驗證要回答的問題
2 週驗證不是為了完成正式研究，而是要回答以下問題：

1. OpenClaw GitHub 首發日是否足以作為研究錨點。
2. 公開財報與公開股價資料是否足以支撐一套穩定的特徵表。
3. 集群分析是否能產出可解釋的公司分群，而不是統計雜訊。

## 3. 驗證範圍上限
為了控制風險，2 週 prototype 不應超過以下範圍：

- 樣本：12-15 家
- 特徵欄位：6-8 個
- 市場分析：1 套事件窗
- 集群分析：1 個主版本
- 文本分析：僅供代表案例人工輔助，不作主體

## 4. 執行順序
### 第 1-2 天
- 確認 OpenClaw GitHub 首發日期
- 鎖定 12-15 家驗證樣本
- 完成欄位字典與資料來源規則

### 第 3-6 天
- 蒐集股價資料
- 蒐集財務資料
- 建立缺值與替代欄位紀錄

### 第 7-9 天
- 完成事件窗市場衝擊指標
- 進行一次集群分析
- 檢查分群是否具備可解釋性

### 第 10-12 天
- 比較各群的事件後表現
- 歸納 2-3 類代表公司輪廓
- 記錄正式版應保留與刪除的方法

### 第 13-14 天
- 撰寫 feasibility memo
- 明確界定正式報告的範圍、限制與替代方案

## 5. 正式報告最低交付標準
- 明確的事件定義
- 清楚的樣本與資料來源規則
- 一套可重現的市場衝擊計算方式
- 一套可解釋的公司分群結果
- 代表公司案例與限制說明

## 6. 明確不建議在第一版納入的內容
- 深度情緒分析
- 多組事件窗同時比較
- 過多分群方法或模型競賽
- NDR、CAC 等揭露不一致欄位
- 任何暗示嚴格因果識別的寫法

## 7. 失敗條件與退場機制
若 2 週內出現以下狀況，正式版應調整為「衝擊排序 + 特徵描述」，不要硬做集群分析：

- 事件日前後市場反應不具可辨識性
- 公司資料缺值過多，無法形成穩定特徵表
- 集群結果在商業語境上無法解釋

## 8. 下一步
下一個實作里程碑應是建立一份 `prototype data sheet`，其中至少包含：

- 公司基本資料
- 事件窗股價欄位
- 6-8 個公開財報特徵欄位
- 缺值與替代欄位備註
